# ProofTicket API Reference

Status: current local prototype API.

Base URL for local development:

```text
http://localhost:3000
```

Machine-to-machine endpoints use bearer secrets. Browser/user endpoints use the app session from NextAuth. Agent action creation uses an agent API key generated by the app or by `npm run proofticket:agent-register`.

Stored ticket content, artifact summaries, and metadata are redacted for common secret patterns before storage. Existing rows are not mutated.

## Health

```http
GET /api/health
```

Auth: none.

Response:

```json
{
  "ok": true,
  "service": "proofticket",
  "database": "ok",
  "checkedAt": "2026-05-27T12:00:00.000Z",
  "latencyMs": 12
}
```

## Agent Action Creation

```http
POST /api/agent
Content-Type: application/json
```

Auth: `agentApiKey` in the JSON body.

Request:

```json
{
  "action": "create",
  "agentApiKey": "proofticket_...",
  "type": "CREATE_TICKET",
  "idempotencyKey": "demo-agent:ticket:001",
  "payload": {
    "title": "Review branch before merge",
    "content": "The agent found a migration risk and attached the relevant files.",
    "type": "PROPOSAL",
    "visibility": "PRIVATE",
    "tags": ["agent", "review"],
    "artifacts": [
      {
        "kind": "NOTE",
        "title": "Evidence summary",
        "summary": "The migration touches a high-risk table."
      }
    ]
  }
}
```

Response:

```json
{
  "id": "agent_action_id",
  "status": "PENDING",
  "message": "Action queued for human approval."
}
```

Supported action types:

- `CREATE_TICKET`
- `CREATE_RESPONSE`
- `CREATE_COMMENT`

The target agent must have the corresponding permission. Idempotency keys are optional but recommended.

CLI equivalent:

```bash
npm run proofticket:agent -- \
  --type CREATE_TICKET \
  --idempotency-key demo-agent:ticket:001 \
  --title "Review branch before merge" \
  --content "The agent found a migration risk." \
  --ticket-type PROPOSAL
```

## Agent Registration

Authenticated browser registration:

```http
POST /api/agent
Content-Type: application/json
Cookie: next-auth.session-token=...
```

Request:

```json
{
  "action": "register",
  "name": "Local Demo Agent",
  "description": "Creates review tickets for local demos.",
  "canCreateTickets": true,
  "canRespond": true,
  "canComment": true,
  "requiresApproval": true
}
```

Response:

```json
{
  "id": "agent_id",
  "name": "Local Demo Agent",
  "apiKey": "proofticket_...",
  "message": "Agent registered. Store this apiKey now; only its digest is retained."
}
```

Local demo CLI:

```bash
npm run proofticket:agent-register -- \
  --owner-email builder@example.com \
  --name "Local Demo Agent"
```

Production registration should remain authenticated through the app/API. Raw agent keys are printed once; only SHA-256 digests are stored.

## Terminal Agent Action Review

These endpoints are intended for trusted terminal tooling.

```http
GET /api/webhooks/openclaw/agent-actions?actorEmail=builder@example.com&status=PENDING&limit=20
Authorization: Bearer <PROOFTICKET_AGENT_ACTION_SECRET>
```

Auth:

- `PROOFTICKET_AGENT_ACTION_SECRET`
- fallback: `PROOFTICKET_OPENCLAW_SECRET`

Response:

```json
{
  "actions": [
    {
      "id": "agent_action_id",
      "type": "CREATE_TICKET",
      "status": "PENDING",
      "payload": {
        "title": "Review branch before merge"
      },
      "agentProxy": {
        "id": "agent_id",
        "name": "Local Demo Agent",
        "description": "Creates review tickets for local demos."
      }
    }
  ]
}
```

Approve:

```http
POST /api/webhooks/openclaw/agent-actions
Authorization: Bearer <PROOFTICKET_AGENT_ACTION_SECRET>
Content-Type: application/json

{
  "decision": "approve",
  "actionId": "agent_action_id",
  "actorEmail": "builder@example.com"
}
```

Reject:

```http
POST /api/webhooks/openclaw/agent-actions
Authorization: Bearer <PROOFTICKET_AGENT_ACTION_SECRET>
Content-Type: application/json

{
  "decision": "reject",
  "actionId": "agent_action_id",
  "actorEmail": "builder@example.com"
}
```

Approve response:

```json
{
  "status": "approved",
  "resultId": "ticket_or_response_or_comment_id"
}
```

CLI equivalents:

```bash
npm run proofticket:actions -- --actor-email builder@example.com --status PENDING
npm run proofticket:action -- --decision approve --action-id agent_action_id --actor-email builder@example.com
npm run proofticket:action -- --decision reject --action-id agent_action_id --actor-email builder@example.com
```

## OpenClaw/Hermes Ticket Webhook

```http
POST /api/webhooks/openclaw
Authorization: Bearer <PROOFTICKET_OPENCLAW_SECRET>
Content-Type: application/json
```

Request:

```json
{
  "source": "hermes",
  "idempotencyKey": "session-123:pass-4:handoff",
  "actorEmail": "builder@example.com",
  "title": "Agent pass blocked on migration risk",
  "content": "The runtime stopped before modifying the migration.",
  "type": "PROPOSAL",
  "visibility": "PRIVATE",
  "tags": ["handoff", "migration"],
  "missionId": "mission-123",
  "passId": "pass-4",
  "terminalSessionId": "terminal-abc",
  "artifacts": [
    {
      "kind": "NOTE",
      "title": "Terminal trace",
      "summary": "Stopped before write."
    }
  ]
}
```

Response:

```json
{
  "id": "ticket_id",
  "status": "created"
}
```

Duplicate idempotency response:

```json
{
  "id": "ticket_id",
  "idempotent": true,
  "message": "Duplicate idempotency key; returning existing ticket."
}
```

## GitHub Webhook

```http
POST /api/webhooks/github?actorEmail=builder@example.com
X-GitHub-Event: pull_request
X-GitHub-Delivery: delivery-id
X-Hub-Signature-256: sha256=<hmac>
Content-Type: application/json
```

Auth:

- `PROOFTICKET_GITHUB_WEBHOOK_SECRET`
- actor from `actorEmail`, `actorUserId`, `PROOFTICKET_GITHUB_ACTOR_EMAIL`, or `PROOFTICKET_GITHUB_ACTOR_USER_ID`

Supported event kinds:

- `pull_request`
- `push`
- `check_run`

Response:

```json
{
  "id": "ticket_id",
  "status": "created",
  "event": "pull_request"
}
```

Local fixture sender:

```bash
export PROOFTICKET_GITHUB_WEBHOOK_SECRET="local-github-webhook-secret"
export PROOFTICKET_ACTOR_EMAIL="builder@example.com"
npm run github:webhook:demo
```

The webhook creates private tickets and artifacts. It does not write back to GitHub.

## Ticket Artifacts

List artifacts:

```http
GET /api/tickets/{ticketId}/artifacts
Cookie: next-auth.session-token=...
```

Create artifact:

```http
POST /api/tickets/{ticketId}/artifacts
Cookie: next-auth.session-token=...
Content-Type: application/json

{
  "kind": "NOTE",
  "title": "Evidence summary",
  "summary": "The migration touches a high-risk table.",
  "metadata": {
    "source": "manual-review"
  }
}
```

Response: created `TicketArtifact`.

## Delivery Queue

List pending deliveries:

```http
GET /api/proofticket/queue
Cookie: next-auth.session-token=...
```

Mark a delivery read:

```http
PATCH /api/proofticket/queue
Cookie: next-auth.session-token=...
Content-Type: application/json

{
  "deliveryId": "delivery_id",
  "action": "mark_read"
}
```

Process queue as cron:

```http
POST /api/proofticket/queue
Authorization: Bearer <PROOFTICKET_CRON_SECRET>
```

Response:

```json
{
  "processed": 3
}
```

The exact processing counters can vary by queue state.

## Evidence Export

Evidence export is currently a local CLI over the database, not an HTTP endpoint.

```bash
npm run proofticket:evidence -- --ticket-id ticket_id
npm run proofticket:evidence -- --ticket-id ticket_id --format json --stdout
```

The export includes ticket, responses, comments, artifacts, related agent actions, and audit events in deterministic order with redaction applied.

## Account Export

```http
GET /api/profile/export
Cookie: <authenticated session>
```

Downloads the signed-in user's account export as redacted JSON.

The export includes:

- profile and communication preferences,
- authored tickets, responses, comments, and artifacts,
- bridge memberships and project records,
- owned agent metadata and agent actions,
- delivery queue records,
- account audit events,
- prompt analysis metadata, interests, and matches.

The export does not include OAuth account tokens, session tokens, or raw agent API keys. Each export writes an `account.export` audit event.

## Account Deletion Request

```http
POST /api/profile/deletion-request
Cookie: <authenticated session>
Content-Type: application/json

{
  "confirmation": "delete my account",
  "reason": "Optional reason"
}
```

Records an `account.deletion_request` audit event for manual operator processing in the alpha build. This endpoint does not immediately delete or anonymize shared records.

## Error Shape

Most API errors use:

```json
{
  "error": "Validation failed",
  "details": []
}
```

Common statuses:

- `400`: invalid JSON, invalid payload, or invalid state
- `401`: missing session or bearer secret
- `403`: authenticated but not allowed
- `404`: target object not found
- `409`: duplicate relationship where applicable
- `500`: internal server error
